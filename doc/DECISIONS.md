# Decisions

## 2026-03-14

- Added the minimal backend API support required by the frontend payment workflow rather than mocking around missing contracts. This keeps the payment form and approval panel aligned with live treasury data and avoids inventing client-only state models for counterparties or approval chains.
- Resolved frontend auth-header handling by reading an existing browser session token when present and attaching it to API requests. This avoids introducing a second auth client while preserving the required Bearer-token behavior for treasury mutations.
- Implemented background processing on top of Supabase/Postgres instead of introducing Redis or Bull. Queue claims are handled through a security-definer RPC that combines `FOR UPDATE SKIP LOCKED` with advisory locking so worker polling stays safe under concurrency.
- Kept retry semantics in application code with deterministic backoff windows and terminal `failed` jobs acting as the dead-letter path. This preserves lightweight infrastructure while keeping retry behavior explicit and testable.
- Stored organization-scope cash-position snapshots with `scope_id = organization_id` instead of `NULL` so deterministic upserts work against the existing uniqueness model without weakening financial history constraints.
- Kept cash-position snapshots in native account currencies and convert them to the organization base currency only at read time. This preserves original-currency auditability while still supporting consolidated liquidity reporting.
- Enriched account listings with latest cash-position and reconciliation state rather than adding another frontend-only account-balance API. This keeps payment initiation, account operations, and dashboard metrics on the same live balance source.
- Persisted AI forecast narrative, prompt context, daily confidence-band data, and accuracy details on `cash_flow_forecasts` / `cash_flow_forecast_lines` instead of leaving them transient in API responses. This keeps the forecasting workflow auditable and allows scenario review without re-running Claude.
- Reused the existing Postgres-backed job queue for forecast generation rather than introducing a second async system. Forecasts longer than 30 days now create a queued placeholder record and are completed by a dedicated `forecast.generate` worker.
- Kept financial forecast math deterministic on the server by recomputing projected net and cumulative balances from Claude-provided inflow/outflow strings, and by deriving confidence bands locally instead of trusting opaque AI arithmetic.
- Added `usage_metrics` as the repository-local cost monitoring table for Anthropic token usage instead of mixing LLM metering into generic audit or notification tables. This keeps spend tracking queryable by org, entity, and model.
- Routed manual statement imports through the existing `bank.sync` worker path so uploaded file content stays ephemeral while the same scheduler/worker infrastructure handles parsing, ingestion, and reconciliation.
- Added explicit reconciliation persistence in `transaction_reconciliations` instead of overloading transaction rows with opaque metadata. This keeps audit trails queryable and preserves a clean separation between source transactions and reconciliation decisions.
- Added a `notifications` table for reconciliation exceptions so unmatched bank activity produces visible treasury-team follow-up items rather than silent status changes.
- Implemented Treasury Copilot as a read-only orchestration layer over the existing treasury services rather than introducing write-capable AI actions. The copilot can explain approval steps and surface live data, but it never executes treasury mutations.
- Stored copilot conversations in `copilot_sessions` with AES-GCM encrypted message payloads inside the persisted JSONB envelope and token-usage metadata at the session level. This keeps chat content encrypted at rest while still allowing compliant session recall and audit-trigger coverage on the parent row.
- Used SSE over the existing backend API routes for chat delivery so users see streamed assistant output and visible tool-call status without adding a separate websocket service.
- Implemented reporting aggregation in Postgres SQL functions instead of duplicating treasury math in frontend or API handlers. This keeps report generation deterministic, close to the audited data model, and reusable for both JSON and CSV delivery paths.
- Kept compliance report downloads regenerable from stored period/report metadata instead of persisting duplicate artifact blobs for every request. The compliance report row remains the immutable archive handle while downloads are re-derived from the same SQL reporting contract.
- Gated `/admin` and `/admin/audit-logs` server-side in the Next.js app using the backend `auth/me` permission model, while the sidebar applies the same permission model client-side for navigation visibility.
- Implemented sweep idempotency by checking `transaction_source_events` for a rule/day execution key rather than creating a separate sweep-run table. This keeps duplicate prevention close to the immutable transaction record path.
- Added `sweeping_rules.rule_name` and `sweeping_rules.max_transfer` through a forward migration to reconcile the existing schema with the liquidity feature contract instead of keeping those fields as transient API-only metadata.
- Kept new intercompany loans in `proposed` status with an `approval_state` projection of `pending_bilateral_approval` because the current schema does not provide a bilateral activation approval ledger.
- Used raw intercompany entity UUID inputs in the frontend create-loan form because the repository currently has no legal-entity registry table to drive a safer selector.
- Kept risk calculation async by persisting the latest `risk_exposures` snapshot and `risk_alerts` records for the dashboard, while `POST /risk/exposures/recalculate` only enqueues `risk.recalculate`. This avoids synchronous policy math in the UI path and gives the dashboard a stable “last calculated” state.
- Implemented organization broadcasts as per-user notification fan-out in the service layer instead of storing unread state on shared `user_id IS NULL` rows. This preserves correct per-user read/unread counts while keeping the nullable column available for future true-broadcast cases.
- Reused the existing Postgres-backed job queue for notification email/webhook fan-out and 90-day retention cleanup instead of adding synchronous delivery or a second scheduler. This keeps delivery side effects auditable and failure handling consistent with other background treasury workflows.
- Subscribed the frontend notification client to targeted `notifications` inserts and deduplicated toast IDs in session storage so reconnects or tab-focus changes do not replay the same alert toast multiple times.
- Stored policy-engine DSL rules under `treasury_policies.rules.dsl` instead of replacing `rules` with a raw array. This keeps the new policy builder compatible with existing risk and forecasting code that still deep-merges object-shaped treasury policy payloads.
- Persisted policy warnings on `payments.notes` and echoed the structured `policy_warnings` array only on the create response. This preserves audit-visible operator context in the row while still giving the payment form enough data to render immediate escalation banners and policy links.
- Resolved `require_approval` workflow selection by matching active payment approval workflows on `approval_workflows.conditions.policyRuleId`, `policyRuleIds`, or generic `source/policyAction` markers before falling back to the default active payment workflow.
